# Redmine 運用設計書（確定版）

最終更新日: 2026-03-01  
目的: ワークフロー管理の最適化（工数管理は別ツールで実施）


# 1. 基本方針
- Redmineはワークフロー管理専用
- 工数管理は別ツールで実施
- 管理作業を増やさない
- 構造は固定型
- テンプレートプロジェクトで標準化


# 2. トラッカー構成

## ① 工程（親チケット）
役割：工程単位の進行管理  
工程は固定。

### 親チケット（工程単位）
- 設計
- 手配
- 組立
- 評価
- 報告

親＝工程管理用  
子の進捗を集約する。


### ステータス
- 新規
- 進行中
- 承認待ち
- 差戻し
- 完了
- 中止


### 遷移ルール

#### 一般メンバー
- 新規 → 進行中
- 進行中 → 承認待ち
- 差戻し → 進行中
- 完了へは変更不可

#### 承認者
- 承認待ち → 完了
- 承認待ち → 差戻し
- 完了 → 進行中（例外的再開）


### 運用ルール
- 承認待ち時に担当者を承認者へ変更
- 差戻し時はコメント必須
- 親は工程完了時にクローズ


## ② 作業（子チケット）
役割：工程内の実作業管理  
親に必ず紐付ける（固定構造）。

### 子チケット例
- 図面修正
- 部品選定
- 評価試験①
- 不具合解析


### ステータス
- 新規
- 進行中
- 完了
- 中止


### 遷移
- 新規 → 進行中 → 完了
- どこからでも中止可

承認待ちは設けない。極力シンプル運用。

## ③ 不具合（独立トラッカー）
役割：  
一次対応と製品ライフ中管理を両立。
工程・作業とは別軸で管理。

### ステータス
- 新規
- 解析中
- 対策中
- 承認待ち（必要時）
- 解決
- クローズ
- 再発

### 基本フロー
新規  
→ 解析中  
→ 対策中  
→ 解決  
→ クローズ  


### 再発フロー
クローズ  
→ 再発  
→ 解析中  

# 3. 親子構造（固定型）
- 工程トラッカーが親
- 作業トラッカーが子
- 不具合は独立管理
- 原則この構造を崩さない


# 4. バージョン管理
用途：開発ゲート管理

例：
- G1: 企画完了
- G2: 設計完了
- G3: 試作完了
- G4: 量産承認

進捗確認はバージョン単位で実施。


# 5. 設計思想
- 承認プロセスは工程のみ
- 作業は軽量管理
- 不具合は再発管理を重視
- 例外を増やさない
- 運用後に見直す
